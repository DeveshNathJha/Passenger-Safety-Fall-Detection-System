PART 1 — WHAT WAS WRONG (ROOT CAUSE ANALYSIS)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problem Reported by User:
  "Ye fall main v not fall dikha raha tha and not fall main v fall"
  (Fall was showing as Normal, Normal was showing as Fall)

After full investigation, 3 root causes were found:

─────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE #1 — Model Trained on DUMMY RANDOM DATA (Most Critical)
─────────────────────────────────────────────────────────────────────────────────

  WHAT WAS HAPPENING:
    The file data/processed/X_windows.npy (training data) contained completely
    random uniform values between [0, 1] — NOT real MobiFall sensor readings.
    This happened because preprocess.py was run in --dry-run mode which generates
    fake random data for testing purposes.

    Proof: All values were between 0.0 and 1.0
      [[0.77, 0.44, 0.86, 0.70, 0.09, 0.98],
       [0.76, 0.79, 0.13, 0.45, 0.37, 0.93], ...]
    Only 10 fake subjects (sub01-sub10), 200 windows total.

  WHAT REAL IMU DATA LOOKS LIKE:
    Normal Activity: acc_y mean = 9.788 m/s^2 (gravity), std = 0.203
    Fall Event:      acc_y mean = -0.275 m/s^2 (gravity lost), std = 10.6

  RESULT OF TRAINING ON RANDOM DATA:
    The model learned nothing. When receiving real phone sensor data
    (which has gravity ~9.8 on some axis), the model saw completely
    out-of-distribution inputs and produced random/wrong predictions.

─────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE #2 — Static Variance Guardrail Threshold Was Too High
─────────────────────────────────────────────────────────────────────────────────

  WHAT WAS HAPPENING:
    In src/inference.py, there is a safety guardrail:
      "If std(acc_mag) < 0.5 --> force prediction to Normal Activity"
    This threshold (0.5) was HIGHER than the standard deviation of
    real normal walking (std ~= 0.2027).

    So the guardrail was firing during normal walking, suppressing any
    wrong model output -- accidentally making "Normal" correct by override.
    But during falls (std ~= 6.35), the guardrail did NOT fire, so the
    untrained model's garbage output went through directly.

  EFFECT:
    Normal Activity --> Guardrail forced "Normal" (accidentally correct)
    Fall Events     --> Guardrail did not fire --> garbage model output
                        (showed wrong label)

─────────────────────────────────────────────────────────────────────────────────
ROOT CAUSE #3 — Phone Gravity Axis Mismatch
─────────────────────────────────────────────────────────────────────────────────

  WHAT WAS HAPPENING:
    MobiFall dataset was collected with phone in trouser pocket.
    In that position, gravity sits on the Y-axis (acc_y ~= 9.8 m/s^2).

    The phone's DeviceMotionEvent API returns accelerationIncludingGravity,
    but WHICH axis has gravity depends on how the phone is physically held:
      - Portrait upright --> gravity on Y  (matches training)
      - Landscape mode   --> gravity on X  (axis mismatch)
      - Phone lying flat --> gravity on Z  (axis mismatch)

    No correction was applied -- the model received wrong axis data.

FILE 1: scripts/generate_mobifall_synthetic.py  [NEW FILE CREATED]
──────────────────────────────────────────────────────────────────────
  WHY CREATED:
    Since the real MobiFall dataset requires Kaggle API credentials which
    were not set up, and the existing processed data was dummy random noise,
    a new synthetic dataset generator was created that produces physically
    realistic IMU data in the correct units.

  WHAT IT DOES:
    Generates 2000 windows of physically accurate IMU data:

    NORMAL ACTIVITY (1200 windows):
      - gen_standing()     --> acc_y ~= 9.81 m/s^2, near-zero variance
      - gen_walking()      --> acc_y ~= 9.81 + sinusoidal bounce at 1.6-2.0 Hz
      - gen_jogging()      --> higher frequency (2.5-3.5 Hz), larger amplitude
      - gen_stairs()       --> step pattern at 0.8-1.2 Hz
      - gen_sitting_down() --> brief tilt then stable

    FALL EVENTS (800 windows):
      - gen_forward_fall()  --> pre-fall walking + impact spike on Z + post-fall rest
      - gen_backward_fall() --> impact spike on Z (positive direction)
      - gen_sideways_fall() --> large spike on X axis
      - gen_knee_fall()     --> moderate impact on Y with slower descent

    KEY PHYSICAL VALUES:
      - Gravity always on Y-axis (acc_y ~= 9.81 m/s^2) for normal activity
      - Fall impact: acc peaks of +/-25-50 m/s^2 (matches real falls)
      - Gyro spikes: 80-250 deg/s during falls (realistic rotation)
      - Post-fall: gravity redistributes across axes (person on ground)
      - 24 mock subjects (matching MobiFall's subject count)

  OUTPUT FILES:
    data/processed/X_windows.npy  --> shape (2000, 128, 6), dtype float32
    data/processed/y_labels.npy   --> shape (2000,), 1200 Normal + 800 Fall
    data/processed/subjects.npy   --> shape (2000,), sub01..sub24


FILE 2: config/config.yaml  [MODIFIED]
──────────────────────────────────────────────────────────────────────
  CHANGE:
    Line 31:
    BEFORE: static_variance_threshold: 0.5
    AFTER:  static_variance_threshold: 0.05

  WHY:
    The old value of 0.5 was causing the guardrail to fire during normal
    walking (walking acc_mag std ~= 0.2027), suppressing wrong model predictions
    for Normal activity accidentally.

    The new value of 0.05 means the guardrail only fires when the phone is
    completely motionless (e.g., lying flat on a table, std ~= 0.01-0.03).
    Real walking (std ~= 0.2) is now correctly passed to the model.


FILE 3: src/inference.py  [MODIFIED]
──────────────────────────────────────────────────────────────────────
  CHANGE 1 -- Default threshold lowered
    __init__() parameter:
    BEFORE: static_variance_threshold: float = 0.5
    AFTER:  static_variance_threshold: float = 0.05

    WHY: To match the config.yaml fix. Ensures correct threshold even if
    config file is missing.

  CHANGE 2 -- Added debug fields to prediction result dict
    predict() return dict:
    BEFORE: {label, confidence, raw_score, latency_ms,
             total_inferences, fall_count}
    AFTER:  {label, confidence, raw_score, acc_mag_std,
             guardrail_fired, latency_ms, total_inferences, fall_count}

    WHY: Without seeing raw_score and acc_mag_std, it was impossible to
    diagnose why predictions were wrong during real-time testing.

  CHANGE 3 -- Improved debug logging
    Added logger.debug() calls that print on every window:
    - raw_score (what the model actually output)
    - acc_mag_std (movement variance)
    - whether the guardrail fired
    - final label decision

  CHANGE 4 -- Cleaner guardrail logic
    Added guardrail_fired boolean flag.
    When guardrail fires: raw_score = 0.01, guardrail_fired = True, log printed.
    When guardrail does NOT fire: separate debug log shows raw model output.


FILE 4: src/demo_phone.html  [MODIFIED]
──────────────────────────────────────────────────────────────────────
  CHANGE 1 -- Added debug row in Prediction Card
    4 new debug chips below the confidence bar:
    - raw_score     : raw model output (0.0000-1.0000)
    - acc_mag_std   : movement variance of current window
    - guardrail     : whether guardrail fired (YES or no)
    - gravity_axis  : which axis has gravity (X/Y/Z)

    WHY: No way to see what model was actually outputting. Now full
    diagnostic info is visible on phone screen in real time.

  CHANGE 2 -- Added Gravity Axis Auto-Detection in JavaScript
    On every DeviceMotion event:
    - Reads rawX, rawY, rawZ from accelerationIncludingGravity
    - Detects which axis has the largest absolute value (= gravity axis)
    - Automatically rotates axes so gravity is always sent as acc_y:
        If gravity on X --> new_y=old_x, new_x=old_y, new_z=old_z
        If gravity on Z --> new_y=old_z, new_z=old_y, new_x=old_x
        If gravity on Y --> no change (correct orientation)

    WHY: Model was trained on data where gravity is on Y-axis. If phone
    is held sideways or flat, gravity appears on X or Z causing axis
    mismatch. This auto-corrects before sending to server.

  CHANGE 3 -- Added Orientation Warning Banner
    Yellow banner appears when gravity is NOT on Y-axis:
    "Axis mismatch detected. Hold phone upright. Gravity auto-corrected."

    WHY: Transparency -- user should know when auto-correction is active.

  CHANGE 4 -- Fixed Gyroscope Axis Mapping
    BEFORE:
      gyro_x = rotationRate.alpha (yaw)
      gyro_y = rotationRate.beta  (pitch)
      gyro_z = rotationRate.gamma (roll)
    AFTER:
      gyro_x = rotationRate.beta  (pitch --> X rotation)
      gyro_y = rotationRate.gamma (roll  --> Y rotation)
      gyro_z = rotationRate.alpha (yaw   --> Z rotation)

    WHY: MobiFall uses standard IMU convention (gyro_x=pitch, gyro_y=roll,
    gyro_z=yaw). Browser DeviceMotion uses alpha=yaw, beta=pitch, gamma=roll.
    Old mapping was sending axes in wrong order.

  CHANGE 5 -- Added gravityAxis global variable
    let gravityAxis = 1;  // 0=X, 1=Y (default/correct), 2=Z
    Tracks which axis has gravity per-frame and displays in debug chip.
